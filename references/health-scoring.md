# 健康评分体系

总分 100 分，6 个维度。确定性检测占 50 分，LLM 辅助检测占 50 分。

评分在 Deep 模式下执行，Quick 模式跳过。

## 维度详情

### 1. 完整性（20 分，LLM 辅助）

检查知识矩阵中每种类型是否有对应文件和内容。

**评分规则**：
- 知识矩阵中的 7 种类型，每种有对应文件 → 每种得 ~3 分
- 关键产出物有版本记录 → 加 2 分
- 关键决策被记录 → 加 2 分（缺一个扣 1 分）
- 无孤儿文件（所有文件被索引引用）→ 加 1 分

**CLAUDE.md 质量检查**（如果有编码项目）：
- CLAUDE.md 中是否记录了必要的 commands/workflows（构建、测试、部署）→ 缺少扣 2 分
- CLAUDE.md 中是否描述了项目结构（Architecture）→ 缺少扣 2 分
- CLAUDE.md 中是否记录了非常规做法和 gotchas → 缺少扣 1 分

**扣分项**：
- 某类知识完全缺失 → 扣 3 分
- 重要产出物无版本记录 → 扣 2 分
- 关键决策未被任何文件记录 → 扣 2 分

#### 知识缺口报告（完整性子步骤）

在评分完成后，对 7 种知识类型做逐类覆盖率统计。按 knowledge-matrix.md 中的类型定义，扫描所有知识文件（memory/, CLAUDE.md, docs/, context/），将每个条目归入对应的类型，统计条目数量。

**输出格式**（附在健康评分报告末尾）：
```
📋 知识缺口报告:
  ✅ 领域知识 (domain): 5 条
  ✅ 项目决策 (decision): 8 条
  ⚠️ 工具经验 (tool): 1 条 — 考虑记录常用工具的使用经验
  ✅ 输出索引 (output): 12 条
  ⚠️ 流程标准 (workflow): 0 条 — 考虑建立项目级操作规范
  ✅ 个人洞察 (insight): 3 条
  ✅ 编码知识 (code): 4 条
```

**缺口判定标准**：
- 0 条 → ⚠️ 缺口，附建议
- 1-2 条 → 🔍 偏少，可关注
- 3+ 条 → ✅ 正常

**建议生成规则**：根据项目类型（编码/内容/混合）和已有知识分布，生成针对性建议。如：编码项目缺工具经验 → "考虑记录开发环境配置和调试工具用法"；内容项目缺流程标准 → "考虑建立内容产出和质量检查流程"。

### 2. 时效性（20 分，确定性）

检查每个文件的"最后更新"标记。

**确定性检测方法**：
1. 扫描所有知识文件，查找以下模式提取更新时间：
   - frontmatter 中的 `updated` 或 `date` 字段
   - 文件末尾的 `*最后更新: YYYY-MM-DD*`
   - 文件修改时间（mtime）作为兜底
2. 计算每个文件距今天数
3. 超过 30 天 → 标记为"需审查"

**评分规则**：
- 所有文件 ≤ 7 天 → 20 分
- 有 1-2 个文件 7-30 天 → 16-19 分
- 有文件超过 30 天 → 每个扣 3 分
- 有文件超过 60 天 → 每个额外扣 2 分
- 无任何更新时间标记的文件 → 每个扣 2 分

### 3. 一致性（15 分，LLM 辅助）

检查不同文件之间的信息是否一致。

**检查项**：
- CLAUDE.md 中提到的工具/命令是否与实际一致
- memory/ 中不同文件对同一主题的描述是否矛盾
- 同一概念是否使用统一术语
- 索引文件中的描述是否与实际文件内容匹配

**评分规则**：
- 无矛盾 → 15 分
- 有 1 处矛盾 → 扣 5 分
- 有 2-3 处矛盾 → 扣 8 分
- 有 4+ 处矛盾 → 扣 12 分
- 术语不统一（同一概念多种叫法）→ 扣 3 分

### 4. 精简性（15 分，确定性）

检查知识文件是否精简、无膨胀。

**确定性检测方法**：
1. 统计 MEMORY.md 行数（`wc -l`，阈值 200 行）+ 字节数（`wc -c`，阈值 25KB）
2. 统计每个记忆文件行数（阈值：100 行）
3. 统计总知识体积（所有 .md 文件的 KB 总和）
4. 对比 curate-history.jsonl 中的历史体积数据

**评分规则**：
- MEMORY.md 行数（悬崖下方梯度）：≤100 行 → 5 分；100-150 → 4 分；150-200 → 3 分
- **硬门（覆盖上方梯度）**：MEMORY.md >200 行 或 >25KB（`wc -c`）→ 精简性直接 **0 分** + Step-0 必修（超出部分 Claude Code 会话开始静默截断=等于没记，非扣分级问题）
- 所有文件 ≤ 100 行 → 5 分；有 1 个超标 → 3 分；有 2+ 超标 → 1 分
- 总体积趋势持平或下降 → 5 分；增长 ≤ 20% → 3 分；增长 > 20% → 1 分
- **体量倒挂（Deep 弱信号）**：memory 总量 > docs+context+CLAUDE.md 总量 → 扣 2 分并标记"毕业债务（稳定知识未上移）"，喂 Step 5 晋升评估

### 5. 可操作性（15 分，LLM + 文件检查）

检查记忆条目是否包含具体行动建议。

**检查项**：
- 记忆条目是否包含"怎么做"而不仅是"是什么"
- 链接是否有效（文件是否存在）
- 命令是否可执行（路径是否正确）
- 配置项是否有具体值（而非"参见文档"）

**评分规则**：
- 大部分条目可操作 → 12-15 分
- 有条目仅描述不指导 → 每个扣 2 分
- 有断裂链接 → 每个扣 3 分
- 有无法执行的命令 → 每个扣 2 分

### 6. 组织性（15 分，确定性）

检查文件组织是否规范。

**确定性检测方法**：
1. 检查 frontmatter 使用是否统一（有或无，不能混用）
2. 检查索引文件是否覆盖所有子文件
3. 检查是否有孤儿文件（未被索引引用的 .md）
4. 检查文件命名是否一致（小写 + 连字符 vs 驼峰 vs 混用）

**评分规则**：
- frontmatter 统一 → 4 分；混用 → 1 分
- 索引覆盖所有文件 → 4 分；有遗漏 → 2 分；无索引 → 0 分
- 无孤儿文件 → 4 分；有孤儿 → 每个扣 1 分
- 命名一致 → 3 分；不一致 → 1 分

## 评级

| 等级 | 分数 | 含义 |
|------|------|------|
| A | 90-100 | 知识体系非常健康，可作为标杆 |
| B | 70-89 | 基本健康，有改进空间 |
| C | 50-69 | 需要关注，部分区域需治理 |
| D | 30-49 | 知识体系较混乱，需要重点治理 |
| F | 0-29 | 知识体系严重缺失或过时 |

## 趋势追踪

从 `curate-history.jsonl` 中取最近一次 Deep 模式的 health_score，与本次对比：

```
趋势: ↑6 (上次72分, 2026-04-20)  → 知识体系在改善
趋势: ↓3 (上次78分, 2026-04-15)  → 知识体系在退化
趋势: →0 (首次评分)               → 无历史数据
```
