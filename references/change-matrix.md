# 编码变更影响矩阵

适用于编码相关的变更，判断代码变更后哪些知识文件需要同步更新。

## 四点补齐规则（红线）

新增一个编码能力时，文档变更**必须四处都补齐**，少一处就算漏改：

| # | 文档 | 写什么 | 面向谁 |
|---|------|--------|--------|
| 1 | **integration-guide** 或对应"外部视角"文档 | **怎么用**（curl / SDK 示例 / 错误码表） | 下游接入者 |
| 2 | **architecture** | **怎么工作**（数据流、状态机、设计取舍） | 架构审查者 |
| 3 | **runbook** | **怎么运维**（冒烟命令、故障排查、环境变量） | 运维/SRE |
| 4 | **handoff** 或 CHANGELOG | **已完成**（交付状态、版本标记） | 接手者/PM |

这是历次知识同步中最常翻的车——"代码改了但文档只改了一半"。如果项目还没有 docs/ 结构，在 CLAUDE.md 中用对应章节代替。

## 变更类型与段落级映射

### 新增功能 / API

| 需要更新的文件 | 精确到段落/章节 | 更新内容 |
|----------------|-----------------|----------|
| CLAUDE.md | 路由清单 / API 列表 | 新增路由路径、方法、参数 |
| docs/integration-guide | Routes 章节 | 加 curl/SDK 示例、错误码 |
| docs/architecture | Routes / Data Model 章节 | 加数据流、状态机 |
| docs/runbook | 相关运维章节 | 加冒烟命令、健康检查 |
| docs/handoff 或 CHANGELOG | 已完成清单 | 标记交付状态 |
| README.md | 功能列表、使用示例 | 新功能说明 |
| memory/ | — | 工具使用经验或踩坑记录 |

### 新增环境变量

| 需要更新的文件 | 精确到段落/章节 | 更新内容 |
|----------------|-----------------|----------|
| .env.example | 变量区 | 添加变量、注释说明 |
| CLAUDE.md | 环境变量表 | 变量名、类型、默认值、用途 |
| docs/runbook | 环境配置章节 | 部署时如何设置 |
| docs/integration-guide | 配置章节 | 下游需要的变量 |

### 新增数据库表 / 数据结构

| 需要更新的文件 | 精确到段落/章节 | 更新内容 |
|----------------|-----------------|----------|
| CLAUDE.md | 数据模型描述 | 表名、字段、关系 |
| docs/architecture | Data Model 章节 | ER 图、字段说明、索引 |
| docs/runbook | 数据库运维章节 | 迁移命令、回滚方案 |
| memory/ | — | 迁移注意事项 |

### 修改依赖

| 需要更新的文件 | 精确到段落/章节 | 更新内容 |
|----------------|-----------------|----------|
| CLAUDE.md | 依赖版本 / 环境要求 | 版本变更、兼容性说明 |
| README.md | 安装步骤、系统要求 | 新的安装命令 |
| docs/runbook | 部署文档 | 环境配置变更 |

### 删除功能

| 需要更新的文件 | 精确到段落/章节 | 更新内容 |
|----------------|-----------------|----------|
| 所有引用该功能的文件 | 全局搜索引用 | 移除所有引用和描述 |
| CLAUDE.md | 对应章节 | 移除命令、配置、路由 |
| memory/ | — | 删除或归档使用经验 |

### 重命名

| 需要更新的文件 | 精确到段落/章节 | 更新内容 |
|----------------|-----------------|----------|
| 所有引用旧名称的文件 | 全局搜索替换 | 新名称 |
| memory/ | — | 更新记忆中的名称引用 |

## 代码真实性验证

自检时必须验证以下内容与代码实际状态一致：

- CLAUDE.md 中提到的**路径**在代码中真实存在
- CLAUDE.md 中提到的**命令**可以实际执行
- CLAUDE.md 中提到的**工具**已安装在当前环境
- CLAUDE.md 中提到的**环境变量**在 .env.example 或实际配置中有定义
- README.md 的安装/运行步骤与代码一致

验证方法：对 CLAUDE.md 中的每个路径/命令/变量，用 `ls`、`grep`、`which` 等命令确认存在。不可执行的标记为"需验证"。

## 编码专属自检项

在 Quick/Deep 模式第四步自检中，如果是编码项目，额外检查：

- [ ] **API 路由完整性**：新增的每个 API 路由，在 integration-guide 和 architecture 都出现了
- [ ] **环境变量完整性**：新增的每个环境变量，在 runbook 和项目根 markdown（CLAUDE.md 或 README）都出现了
- [ ] **数据模型完整性**：新增的每个数据库表/数据结构，在 architecture 的 Data Model 和项目根 markdown 都出现了
- [ ] **四点补齐**：新增能力的文档变更，integration-guide、architecture、runbook、handoff/CHANGELOG 四处都补了
- [ ] **代码真实性**：CLAUDE.md 提到的路径/命令/工具/环境变量在代码中真实存在

## 跨项目影响（⚠️ 高频翻车点）

**这是历次知识同步中最常遗漏的场景。**

当代码变更涉及共享库、SDK、API 接口时：

1. **识别下游**：哪些项目依赖此项目？（通过 SDK、API、子域、环境变量、npm/pip 包）
2. **双边对齐**：上游改了 → 下游项目的 CLAUDE.md 和 docs/ **也要改**
3. **检查清单**：
   - 下游项目的 CLAUDE.md 是否引用了变更的接口 → 需同步
   - 下游项目的 integration-guide 是否描述了旧接口 → 需更新
   - 下游项目的环境变量是否依赖上游 → 需检查
4. **审计记录**：在审计日志中记录跨项目影响

**关键检查**：本次对话是不是跨项目的？如果是，**两个项目的 docs 都要对齐**。不要假设一个项目改了另一个就不用改。

## 使用方式

在 Curate 的第二步（识别会话变更）中，当判断变更为编码相关时，读取本文件并按矩阵检查对应的文件和段落是否需要更新。
